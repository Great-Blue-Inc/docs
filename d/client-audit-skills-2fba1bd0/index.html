01

What the toolkit is

Shipped Sandbox-validated

Claude Skills that read a Salesforce org's metadata and hand back a client-presentable audit, without a human clicking through Setup for a week.

The idea high level

A manual Salesforce health review is 30 to 40 hours of clicking through Flows, validation rules, profiles, packages, reports, and field usage. This toolkit reads that same metadata directly and drafts the findings, so the reviewer spends their time judging results instead of gathering them. The README frames it honestly as a multi-day, mostly-automated process, not a one-click miracle. The real time cost is validating each skill against a live org, which is hours rather than weeks.

Everything in v1 is read-only metadata. It reads Flow XML, validation rules, profiles and permission sets, installed packages, report and dashboard usage, and aggregate field usage. It never writes, and it never reads raw record values.

Nothing here has run against a real paying client yet. "Validated" means the mechanics work and generalize across three sandboxes with real client metadata, not that it is invoice-ready. Two guardrails from the README hold everywhere: never share skill source or scripts with clients, only the final report, and validate each skill against a live engagement before generalizing it.

02

The audit skills

10 shipped 1 descoped

Ten focused skills, each reading one slice of metadata, plus an orchestrator that chains them into one report. One skill is deliberately held back for v1.

What each one covers high level

Every skill answers one question a reviewer would otherwise dig for by hand, and every finding it produces carries an insight, not just a raw list. The orchestrator runs them together and renders a single summary with one drill-down page per area.

SkillReadsWhat it surfaces
audit-flow-catalogFlow XMLEvery Flow with purpose, trigger, key logic, and risk flags. No Flow Builder clicking.
audit-validation-rulesValidation rule metadataEvery rule and stage-advancement or required-field gate, plus stale or conflicting ones.
audit-automation-summaryWorkflow + Process metadataLegacy Workflow Rules and Process Builder still active, overlaps with Flows, migration candidates.
audit-field-usageAggregate field dataPopulation percentage per field across core objects. Which fields are live vs dead weight.
audit-field-hygieneField metadataUnused fields and naming-convention violations across core and custom objects.
audit-report-hygieneReport + dashboard usageStale reports and dashboards not opened in a window (default 12 months) as deletion candidates.
audit-installed-packagesInstalled package metadataPackages with versions, out-of-date flags, best-effort usage cross-reference.
audit-integration-usersProfiles, perm sets, usersIntegration users and their permission hierarchy. Elevated, orphaned, over-permissioned accounts.

audit-storage-trendStorage usageYear-over-year file and data storage growth as a trend.
audit-report-orchestratorAll of the aboveChains every audit skill and renders one client HTML summary plus a drill-down page per area.

The one held back low level

Descoped for v1 · decision 2026-06-30

audit-duplicates flags likely duplicate Contacts and Leads. It is the one skill that would touch actual record data rather than metadata, so it needs real script engineering. Per the README, that is "not worth blocking the rest, which are genuinely hours-not-weeks work." It is descoped, not deleted. The file is kept as a design record with stricter rules: a deterministic script and aggregate-only output (group counts plus record IDs, never raw PII).

03

The org-identity safety gate

Sandbox-default

Before any skill runs, the toolkit proves which org it is pointed at and refuses to touch production without an explicit go.

Why it exists high level

These skills read client orgs. Running the wrong one, or running against production by accident, is the failure that matters most. So every run starts by stating the org identity out loud and stopping cold if it is not a sandbox.

The gate, exactly low level

Both commands run first, and their results are recorded in the run notes:

# 1. identity: record username + instanceUrl
sf org display --target-org <alias> --json

# 2. sandbox check: record Name + IsSandbox
sf data query --query "SELECT Name, IsSandbox FROM Organization" \
  --target-org <alias> --json
The stop rule (quoted)

"State the org Name and IsSandbox in the run notes. If IsSandbox=false, STOP and get explicit approval before proceeding. These skills default to sandbox-only." A non-zero exit code or a "status" != 0 in any sf --json output is treated as a failed step, and partial output is never parsed.

04

The report and dossier output system

v4 live

A run ends in one self-contained HTML report plus a drill-down page per area, in the same house dossier style as this document, with zero external requests.

What a client sees high level

Report template v4, adopted 2026-07-06, is the Great Blue engineering dossier system, the same identity you are reading now, ported into the audit report generator. It is a restrained token system with no ornament. It replaced a minimal v3 template and is not the earlier rejected ornate style. The report opens as a single HTML file, renders in light and dark, and works offline with no fonts or scripts fetched from the network.

How a run becomes a report low level

Each run writes a run.json data file, and one parameterized generator turns it into the HTML. The schema has been stable across v3 and v4:

org_alias · client_name · date
Run identity, echoed into the masthead.
title · dek
Report headline and standfirst.
stats
List of [value, label] pairs, rendered as metric tiles.
pop
Dict of population and bucket-chart context.
findings
List of 10 objects, one per audit area, each with slug, title, sev, sev_label, metric, metric_label, body, and detail.

The build pipeline

  • scripts/generate_report.py is the single generator. Everything below its template marker is the dossier token system.
  • Output is standalone HTML with system font stacks only, a full doctype, charset, viewport, and a data-URI favicon. No webfont link, no CDN. This fixed a real v3 defect where a Google Fonts dependency fell back silently from file://.
  • Light and dark both, via token-level prefers-color-scheme plus a data-theme override and a working per-page toggle, drill-downs included.
  • Severity pills are derived from each skill's own flags: teal for clean, amber for watch, rose for risk, neutral for limited.
  • One drill-down page per audit area, and every table carries an insight column rather than a bare identifier dump.
  • html.escape() runs on every run.json string, with one whitelisted exception for bare <code> spans.

Verification and output locations

A Playwright headless harness, scripts/verify-ui.mjs, screenshots the report over file://. v4 added a --colorScheme flag so both themes are verified. Outputs live in predictable places:

PathHolds

In progress and queued

  • Pop-chart bucket severity class is escaped but not yet validated against the allowed set.
  • Two legacy per-org generator forks are superseded and pending a delete decision.
  • The audit-report-hygiene query was revised 2026-07-02 and needs a sandbox re-test.
Repo wiring · as of 2026-07-08

Dual-push is not wired on the working clone. The README describes a single git push writing both br5692 (primary) and Great-Blue-Inc (mirror), but this clone's origin points only at Great-Blue-Inc, and the mirror was last pushed by explicit URL. Running /dual-push in the repo restores the single-push-hits-both setup. Worth doing before the next round of commits so the two remotes do not drift.

Descoped
